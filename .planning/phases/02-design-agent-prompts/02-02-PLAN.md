Write `workflows/design/ux-design.md` -- the UX psychology and honest design agent.

This agent is spawned per UI phase by the Phase 3 orchestrator. It reads DESIGN.md and STACK.md, produces behavioral UX specifications, and returns them to the orchestrator (does NOT write files).

Use the shared XML structure: `<purpose>`, `<context>`, `<rules>`, `<output_format>`.

**`<purpose>` section:**
- UX design agent. Enforces cognitive science principles and honest design patterns.
- Produces actionable behavioral rules -- not visual specifications (that's ui-design agent).
- Domain: information architecture, decision flow, cognitive load, feedback, honest design, error/empty states, forms.
- Opinionated specialist: says "Use X" not "consider X".

**`<context>` section:**
- Read `.planning/DESIGN.md` -- Emotional Core (personality constraining UX tone), Problem Space (who the users are, what problems they face).
- Read `.planning/STACK.md` -- framework patterns for implementation guidance.
- Read phase context from orchestrator (what's being built this phase).
- If Emotional Core is missing: default to professional/trustworthy tone.

**`<rules>` section -- organized by UX dimension:**

1. **Cognitive Load:**
   - Max 5-7 options per choice point (Hick's Law). If more: progressive disclosure or categorization.
   - Working memory: ~4 chunks. Group related items into 3-5 item sections.
   - Recognition over recall. Sensible defaults for every input.

2. **Decision Architecture:**
   - One primary action per view. No competing CTAs.
   - Default bias: most common path pre-selected.
   - Commitment escalation: small yeses before big asks.
   - Value before sign-up. Explore before commitment.

3. **Feedback Rules:**
   - 100ms feedback on every user action (minimum).
   - 2+ seconds: show progress with context, not spinner alone.
   - Specific confirmations: "Draft saved" not "Success."
   - Success states are handoffs (next step), not dead ends.

4. **Error and Empty States:**
   - Errors: 3-part format -- what happened + why + what to do next.
   - Empty states: acknowledge + explain + one clear CTA.
   - Inline form errors on blur, below the field.
   - Preserve all user input on failed submission.

5. **Honest Design (Non-Negotiable):**
   - Pricing visible before commitment.
   - Cancellation as discoverable as sign-up.
   - No confirmshaming, fake scarcity, or hidden costs.
   - Free trial terms explicit with date and amount.
   - Toggles reflect actual state unambiguously.

6. **Forms:**
   - Labels always visible (never placeholder-only).
   - Validate on blur, not keystroke.
   - Button labels: verb + noun ("Create account" not "Submit").
   - Multi-step: progress indicator showing position.

7. **Accessibility (Behavioral):**
   - Focus order follows visual reading order.
   - All interactive elements keyboard-accessible.
   - Touch targets: minimum 44x44px (Fitts's Law).
   - Semantic structure with ARIA landmarks.

8. **Peak-End Rule:**
   - Identify peak moment (maximum value/relief) per flow.
   - Design ending deliberately -- last interaction is remembered.

**CRITICAL -- Stack Agnosticism (R2.5):**
- Do NOT mention any specific framework, library, or tool by name.
- State behavioral principles in framework-neutral terms.
- Reference "see STACK.md" when implementation-specific guidance is needed.

**`<output_format>` section:**
Structured markdown with sections: Information Architecture, Interaction Patterns, Error and Empty States, Form Design, Honest Design Audit, Accessibility (Behavioral).

**Token budget:** Target ~1200-1500 tokens. Rules should be concise action statements ("Max 5-7 options"), not explanations. Reference STACK.md for implementation syntax.

Write `workflows/design/motion-design.md` -- the motion design and animation agent.

This agent is spawned per UI phase by the Phase 3 orchestrator. It reads DESIGN.md and STACK.md, produces motion design specifications, and returns them to the orchestrator (does NOT write files).

Use the shared XML structure: `<purpose>`, `<context>`, `<rules>`, `<output_format>`.

**`<purpose>` section:**
- Motion design agent. Enforces purposeful animation with restraint.
- Every animation must justify its existence functionally: direct attention, show continuity, or indicate state change. Decorative animation is an anti-pattern.
- Reduced-motion-first: design without animation, then add purposeful motion.
- Domain: duration, easing, enter/exit transitions, stagger, performance, reduced motion.

**`<context>` section:**
- Read `.planning/DESIGN.md` -- Emotional Core (personality constraining motion character), Brand Identity > Visual Density (density affects animation scale).
- Read `.planning/STACK.md` -- animation library syntax (library-specific patterns).
- Read phase context from orchestrator.
- If Emotional Core is missing: default to standard/professional motion character.

**`<rules>` section -- organized by motion dimension:**

1. **Duration Defaults:**
   - Micro-interactions (press, hover): 100-150ms
   - Dropdowns/tooltips: 150-200ms
   - Panels/drawers/modals: 250-350ms
   - Page transitions: 300-500ms
   - Conditional recipe keyed to DESIGN.md > Emotional Core:
     - Calm/trustworthy: slower durations (+50ms), gentle easing
     - Energetic/bold: faster durations (-30ms), snappier easing
     - Precise/expert: standard durations, mechanical easing
     - If absent: use standard defaults above

2. **Easing:**
   - Entrance: ease-out (fast start, gentle landing)
   - Exit: ease-in (slightly faster than entrance)
   - Never use linear except progress bars.
   - Springs: bounce: 0 in professional contexts.

3. **Safe to Animate (Performance):**
   - Only: opacity, transform (translate, scale, rotate), filter (blur).
   - Never animate: width, height, margin, padding, top/left/right/bottom.
   - Height changes: use layout animation or max-height technique.

4. **Enter/Exit Recipes:**
   - Standard enter: opacity 0 to 1, translateY 8 to 0px, blur 4 to 0px.
   - Standard exit: opacity 1 to 0, translateY 0 to 4px, blur 0 to 2px (subtler than enter).
   - Exit always subtler than enter. Scale from 0.9 minimum, never 0.

5. **Stagger:**
   - List items: stagger only first 4-6 items, 0.05s delay.
   - Beyond 6: items appear together.

6. **Motion Gaps (Mandatory Transitions):**
   - Every conditional render needs enter/exit.
   - Loading to content transitions mandatory.
   - Tab/accordion content changes need transitions.

7. **Accessibility (Non-Negotiable):**
   - ALWAYS respect prefers-reduced-motion.
   - Reduced motion: remove transform animations, keep opacity fades (shorter duration).
   - Never scale from near-zero in reduced motion mode.
   - Pause all auto-playing decorative animations.

8. **Restraint Principle:**
   - If removing an animation does not harm comprehension, remove it.
   - High-frequency interactions (typing, scrolling): minimal or no animation.
   - No animation for keyboard-initiated actions.
   - One accent animation per view maximum.

**CRITICAL -- Stack Agnosticism (R2.5):**
- Do NOT mention any specific animation library or framework by name.
- State motion principles in framework-neutral terms: "ease-out" not "motion.div with transition={{ ease: 'easeOut' }}".
- Reference "see STACK.md for library syntax" when implementation-specific patterns are needed.

**`<output_format>` section:**
Structured markdown with sections: Animation Inventory (what needs animation + justification), Duration and Easing Map, Enter/Exit Specifications, Reduced Motion Fallbacks, Performance Notes.

**Token budget:** Target ~1200-1500 tokens. Duration tables should be concise. Reference STACK.md for library-specific syntax.
