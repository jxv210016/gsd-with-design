# Phase 5: Auxiliary Commands & Quick Reference - Research

**Researched:** 2026-03-05
**Domain:** Claude Code slash command authoring / design reference output
**Confidence:** HIGH

## Summary

Phase 5 creates two read-only slash commands: `design-ui` and `design-stack`. These are convenience commands that output design craft standards and stack conventions during implementation -- quick reference cards that developers invoke mid-task without leaving Claude Code. The commands do not modify files, run agents, or trigger workflows. They read existing files (DESIGN.md, STACK.md, and the design agent workflow files) and present a consolidated summary.

The work is minimal in scope: two markdown files in `.claude/commands/gsd/` following the exact same frontmatter and structure pattern as the existing `design-thinking.md` command. The primary challenge is content curation -- deciding what subset of the design agent rules to surface in each command for maximum utility during implementation, without duplicating full agent content.

**Primary recommendation:** Each command should read DESIGN.md + relevant workflow files at runtime and output a condensed quick-reference card. Commands should be pure Read-only (no Write, no Bash needed). Content should be organized as actionable checklists and lookup tables, not prose.

## Architecture Patterns

### Command File Structure
Commands live in `.claude/commands/gsd/` and follow this exact pattern (verified from existing `design-thinking.md` and `help.md`):

```
.claude/
  commands/
    gsd/
      design-thinking.md   # (Phase 1 -- already exists)
      design-ui.md         # (Phase 5 -- R5.1)
      design-stack.md      # (Phase 5 -- R5.2)
```

### Pattern 1: Read-Only Reference Command
**What:** A slash command that reads files and outputs formatted reference content. No file writes, no interactive prompts.
**When to use:** When the user needs to look up design rules during implementation.
**Example (from existing `help.md`):**
```markdown
---
name: gsd:help
description: Show available GSD commands and usage guide
---
<objective>
Display the complete GSD command reference.

Output ONLY the reference content below. Do NOT add:
- Project-specific analysis
- Git status or file context
- Next-step suggestions
- Any commentary beyond the reference
</objective>

<process>
Output the complete GSD command reference from @workflow-file.
Display the reference content directly -- no additions or modifications.
</process>
```

### Pattern 2: Context-Aware Reference (design-ui and design-stack)
**What:** Unlike `help.md` which outputs static content, these commands must read project-specific files (DESIGN.md, STACK.md) and merge them with the static craft rules from the workflow files.
**When to use:** When the reference needs to be personalized to the project's brand identity and tech stack.
**Structure:**
```markdown
---
name: gsd:design-ui
description: Quick reference for UI/UX/motion craft standards
allowed-tools:
  - Read
---
<objective>
Read-only output of design craft standards personalized to this project's DESIGN.md.
</objective>

<context>
Read these files:
1. `.planning/DESIGN.md` -- Brand Identity and Emotional Core
2. `workflows/design/ui-design.md` -- spacing, color, typography, states rules
3. `workflows/design/ux-design.md` -- cognitive load, honest design, feedback rules
4. `workflows/design/motion-design.md` -- duration, easing, reduced-motion rules
</context>

<process>
[Output format instructions]
</process>
```

### Pattern 3: Graceful Degradation
**What:** Commands must handle missing files (no DESIGN.md = project hasn't run design thinking yet).
**When to use:** Always -- user may invoke commands before running design-thinking.
**Behavior:**
- No DESIGN.md: output generic craft rules (the agent defaults) with a note to run `/gsd:design-thinking` first
- No STACK.md: output generic CSS translations with a note that stack conventions haven't been generated
- Both present: output personalized reference merged with craft rules

### Anti-Patterns to Avoid
- **Duplicating full agent content:** The commands should summarize and curate, not copy-paste the entire agent prompt files. Key rules and lookup tables only.
- **Writing files:** These are read-only. No Write tool in allowed-tools. No file creation.
- **Interactive prompts:** No AskUserQuestion. These output content and exit.
- **Referencing workflow files with @-syntax:** The `@` file inclusion syntax pulls in content at command load time. For dynamic content that depends on project state (DESIGN.md, STACK.md), use Read tool instructions instead.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Outputting design rules | Hardcoded rules in command file | Read from workflow files at runtime | Rules stay in sync with agent files -- single source of truth |
| Project-specific values | Hardcoded brand values | Read from DESIGN.md at runtime | Each project has different brand identity |
| Stack translations | Framework-specific recipes in command | Read from STACK.md at runtime | Stack varies per project |

**Key insight:** The command files should be thin orchestration layers that read and format content from existing workflow files and planning artifacts. The rules themselves live in the Phase 2 agent files and the generated STACK.md.

## Common Pitfalls

### Pitfall 1: Content Duplication Leading to Drift
**What goes wrong:** Copying design rules into the command file means they drift out of sync with the agent workflow files when agents are updated.
**Why it happens:** Seems simpler to inline the content rather than read files at runtime.
**How to avoid:** Commands read workflow files at runtime and extract/format relevant sections. The workflow files remain the single source of truth.
**Warning signs:** Same design rule appears in both a command file and a workflow file.

### Pitfall 2: Overly Verbose Output
**What goes wrong:** Outputting all design agent content makes the reference card too long to scan quickly.
**Why it happens:** Including everything feels more complete.
**How to avoid:** Curate for quick-reference utility. Use tables and checklists. Focus on lookup values (spacing scale, color ratios, duration defaults) not explanatory prose.
**Warning signs:** Output exceeds what fits in a single screen of terminal output.

### Pitfall 3: Missing Graceful Degradation
**What goes wrong:** Command errors or outputs nothing when DESIGN.md or STACK.md doesn't exist.
**Why it happens:** Assuming the design thinking flow has always been run.
**How to avoid:** Check file existence. Output generic defaults with a note pointing to `/gsd:design-thinking`.
**Warning signs:** Error messages when running command on a fresh project.

### Pitfall 4: Wrong Command Location
**What goes wrong:** Placing commands in `commands/gsd/` (project root) instead of `.claude/commands/gsd/`.
**Why it happens:** PROJECT.md says `commands/gsd/design-*` but the actual working pattern from Phase 1 and Phase 4 uses `.claude/commands/gsd/`.
**How to avoid:** Follow the established pattern from `design-thinking.md` which lives in `.claude/commands/gsd/`.
**Warning signs:** Slash command not discoverable via `/gsd:design-ui`.

## Open Questions

1. **Should commands use @-include for workflow files or Read tool?**
   - What we know: `@` syntax pulls files in at command parse time (static). Read tool loads at runtime (dynamic).
   - What's unclear: Whether Claude Code caches @-included content or re-reads each invocation.
   - Recommendation: Use Read tool instructions in `<process>` for DESIGN.md and STACK.md (definitely dynamic). For the three design agent workflow files, `@` inclusion is acceptable since they are static files that don't change per-project. However, to keep the pattern consistent and the command file small, use Read tool for all files.

2. **How much content from each agent to include in design-ui?**
   - What we know: Each agent is ~800 tokens. All three together would be ~2400 tokens of rules.
   - What's unclear: Optimal reference card length for mid-implementation lookup.
   - Recommendation: Extract only the lookup tables and checklists (spacing scale, color ratios, duration defaults, state checklist, honest design checklist). Skip explanatory prose. Target a scannable reference that fits in one screen.
