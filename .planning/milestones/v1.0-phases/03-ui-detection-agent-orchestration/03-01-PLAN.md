The file must contain these sections:

**purpose:** Explain this is the UI phase detection gate. It reads phase text from ROADMAP.md and optionally CONTEXT.md, returning a boolean IS_UI decision.

**context:** Specify the two text sources to scan:
1. ROADMAP.md phase section text (obtained via `gsd-tools.cjs roadmap get-phase`)
2. CONTEXT.md for the phase (if it exists at `{phase_dir}/{phase}-CONTEXT.md`)

**rules:** The detection algorithm in this exact priority order:

Step 1 — Manual override markers (ABSOLUTE PRIORITY):
- Scan ROADMAP.md phase section for `<!-- ui-phase -->` marker. If found, return IS_UI=true with notice: "UI detection overridden by manual marker: forced UI phase"
- Scan for `<!-- no-ui -->` marker. If found, return IS_UI=false with notice: "UI detection overridden by manual marker: forced non-UI phase"
- Also check for `<!-- design-refresh-stack -->` marker and pass that flag to the orchestrator

Step 2 — Negative keyword suppression (runs before positive matching):
- Check phase section + CONTEXT.md text for negative keywords
- Negative keyword list (use these exact terms): unit test, integration test, migration, CLI, API endpoint, backend, database, schema, model, ORM, queue, worker, cron, webhook, infrastructure, deployment, CI/CD, pipeline, server, microservice, authentication logic, authorization, RBAC, token, certificate
- If ANY negative keyword appears as a dominant term in the phase text, return IS_UI=false
- "Dominant" means: the phase description is primarily about that backend concern, not just mentioning it in passing. For example, "Create user authentication API with JWT tokens" is backend-dominant. "Create login page with form validation" is NOT backend-dominant despite mentioning authentication.

Step 3 — Positive keyword matching across 6 categories (2+ category threshold):
- Define 6 keyword categories with these exact terms:

Components: button, card, modal, dialog, form, input, dropdown, menu, table, list, sidebar, navbar, header, footer, toast, tooltip, tab, accordion, carousel, badge, avatar, checkbox, radio, select, textarea, slider

Layouts: grid, flexbox, layout, responsive, breakpoint, column, row, container, page, screen, view, panel, dashboard, sidebar-layout, split-view

Interactions: click, hover, drag, scroll, swipe, tap, gesture, touch, press, focus, blur, submit, toggle, expand, collapse, sort, filter, search, pagination, infinite-scroll

Visual: color, theme, dark-mode, light-mode, icon, image, animation, transition, gradient, shadow, border, typography, font, spacing, padding, margin, opacity, elevation

Navigation: route, page, tab, breadcrumb, link, navigate, redirect, back, forward, menu, drawer, stepper, wizard, onboarding, flow

States: loading, error, empty, skeleton, placeholder, disabled, active, selected, checked, expanded, collapsed, success, warning, progress, pending

- Count how many DISTINCT categories have at least 1 keyword match (case-insensitive)
- If matched_categories >= 2, return IS_UI=true
- If matched_categories < 2, return IS_UI=false

Step 4 — Conditional DESIGN.md gate:
- When IS_UI=true, instruct the caller to load `.planning/DESIGN.md` for design agent context
- When IS_UI=false, instruct the caller to skip all design artifacts (no DESIGN.md, no agents, no {phase}-UI.md)
- If IS_UI=true but `.planning/DESIGN.md` does not exist, instruct the caller to prompt the user: "UI phase detected but no DESIGN.md found. Run /gsd:design-thinking first, or continue without design guidance?"

**output_format:** Specify that the detection result should be communicated as:
- IS_UI: true/false
- DETECTION_METHOD: "marker-override" | "negative-suppressed" | "keyword-threshold"
- MATCHED_CATEGORIES: [list of matched category names] (only for keyword-threshold)
- REFRESH_STACK: true/false (from marker check)
- Notice text (if marker override was used)

Keep the file concise — target ~120-150 lines matching the style of the existing design workflow files. Use clear section headers within the XML structure. State the algorithm as instruction steps, not pseudocode — this is a prompt for Claude, not executable code.
