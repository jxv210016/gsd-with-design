**Frontmatter:**
```yaml
---
name: gsd:design-thinking
description: Run design thinking interview to create DESIGN.md
argument-hint: ""
allowed-tools:
  - Read
  - Write
  - Bash
  - AskUserQuestion
---
```

**Objective section:** Explain this command runs a conversational design thinking interview producing `.planning/DESIGN.md` with Problem Space, Emotional Core, Solution Space, and Brand Identity sections. Mention it can be run standalone or will be embedded in new-project (Phase 4).

**Process section -- build ALL of the following steps:**

**Step 0: Check existing DESIGN.md**
- Read `.planning/DESIGN.md` -- if it exists:
  - AskUserQuestion (header: "DESIGN.md", max 9 chars, OK) asking "You already have a DESIGN.md. What would you like to do?" with options: "Update -- revise specific parts", "View -- show me what's there", "Replace -- start fresh"
  - If "Update": Display current DESIGN.md, ask "What would you like to change?" as plain text, update relevant sections, enter validation loop (Step 7)
  - If "View": Display DESIGN.md content, then AskUserQuestion offering Update or Replace
  - If "Replace": Continue to Step 1 (full re-interview)

**Step 1: Skip offer (first interaction for new interviews)**
- AskUserQuestion (header: "Design", 6 chars, OK) asking "Design thinking helps ground your project in user needs and emotional direction. Skip to use vanilla GSD, or continue?" with options: "Continue -- let's think about design", "Skip -- I'll use vanilla GSD"
- If "Skip": Output a brief message that design thinking was skipped and no DESIGN.md was created. Exit command. Do NOT write any file. Done.
- If "Continue": Proceed to Step 2.

**Step 2: Load PROJECT.md context**
- Read `.planning/PROJECT.md` if it exists. Extract: project name, description, tech stack, target users, constraints.
- Use extracted context to pre-fill interview questions (e.g., don't ask about tech stack if PROJECT.md already specifies it; instead confirm or adjust).
- If PROJECT.md doesn't exist, proceed with all questions open.

**Step 3: Design thinking interview -- Problem Space**

Guide the user through Problem Space with 2-3 AskUserQuestion calls, each building on prior answers. The conversation should feel like a thinking partner, not a form.

Questions should cover:
- Target Users: Who experiences this problem? (header: "Users", options: pre-filled from PROJECT.md if available, or generic persona categories)
- Core Problem: What specific problem do they face? (header: "Problem") Include inline example -- Good: "Developers waste 30 min per PR reviewing inconsistent UI decisions" vs Bad: "Bad design"
- Pain Points: What's broken about current alternatives? (header: "Pain Points")

If PROJECT.md already captured target users, weave that in: "Your project targets [X]. What specific problem are they facing?" rather than re-asking.

**Step 4: Design thinking interview -- Emotional Core**

This is the most critical section. Users default to vague adjectives without guidance.

- Primary emotional statement (header: "Emotion"): "How should using your product FEEL? Not what it does -- how it makes people feel."
  - MUST show inline examples before asking:
    - Bad: "modern and clean" (too vague -- every product claims this)
    - Good: "calm confidence -- like a trusted advisor who never rushes you"
    - Good: "playful momentum -- like a pinball machine that rewards curiosity"
  - Options should be 2-3 feeling-direction starters (e.g., "Calm and trustworthy", "Energetic and bold", "Precise and expert") -- user refines via "Other"
- Supporting attributes (header: "Attributes"): "What 3-4 words describe the personality behind this feeling?"
  - Options with curated attribute clusters (e.g., "Trustworthy, patient, expert", "Bold, playful, surprising", "Minimal, precise, quiet")

**Step 5: Design thinking interview -- Solution Space**

- Key Capabilities (header: "Capabilities"): What are the 2-3 most important things v1 must do? Plain text follow-up for details.
- Tech Stack (header: "Stack"): If PROJECT.md has tech stack, confirm: "PROJECT.md says [React + Tailwind]. Keep this, or change?" If no PROJECT.md, ask: framework, styling approach, key libraries. Options should include common stacks.

**Step 6: Design thinking interview -- Brand Identity**

- Color Mood (header: "Color Mood"): "What color temperature fits your product's emotional direction?" Options: "Warm (oranges, reds, earth tones) -- inviting, human", "Cool (blues, greens, grays) -- professional, calm", "Neutral (black, white, silver) -- minimal, modern"
- Typography Feel (header: "Typography"): Options: "Geometric (Inter, Helvetica) -- modern, clean", "Humanist (Source Sans, Lato) -- friendly, approachable", "Monospace (JetBrains Mono, Fira) -- technical, precise"
- Visual Density (header: "Density"): Options: "Spacious -- lots of breathing room", "Balanced -- moderate whitespace", "Dense -- information-rich, compact"
- Brand Personality: Plain text question: "How does this brand speak? Confident but not arrogant? Friendly but not casual? And what must it NEVER feel like?"

**Step 7: Generate DESIGN.md and validate**

Assemble the complete DESIGN.md using this exact schema template:

```markdown
---

---

# DESIGN.md

## Problem Space

### Target Users
- {from interview}

### Core Problem
- {from interview}

### Current Alternatives
- {from interview}

### Pain Points
- {from interview}

## Emotional Core

### Primary Emotional Statement
{One sentence -- the feeling-statement from interview}

### Supporting Attributes
- {Attribute 1}
- {Attribute 2}
- {Attribute 3}
- {Attribute 4}

## Solution Space

### Key Capabilities
- {Capability 1}
- {Capability 2}
- {Capability 3}

### Tech Stack
- **Framework:** {from interview or PROJECT.md}
- **Styling:** {from interview or PROJECT.md}
- **Key Libraries:** {from interview or PROJECT.md}

## Brand Identity

### Visual Direction
- **Color Mood:** {warm / cool / neutral}
- **Typography Feel:** {geometric / humanist / monospace}
- **Visual Density:** {spacious / balanced / dense}

### Brand Personality
- {How the brand speaks}
- {Tone description}

### Anti-Patterns
- {What this brand must NEVER feel like}
```

Display the complete DESIGN.md to the user, then enter validation loop:

- AskUserQuestion (header: "Direction"): "Does this capture your direction?" Options: "Yes -- looks good", "Edit -- I want to change something", "Regenerate -- rewrite from my answers"
- If "Yes": Write `.planning/DESIGN.md` using Write tool. Commit with gsd-tools. Done.
- If "Edit": Ask "What would you like to change?" as plain text. User describes changes in natural language. Update relevant sections. Re-display full DESIGN.md. Loop back to validation question.
- If "Regenerate": Keep all interview answers but produce a fresh interpretation of the DESIGN.md (different phrasing, emphasis). Re-display. Loop back to validation question.
- Loop is unlimited -- continue until user selects "Yes".

**Success criteria section:**
- DESIGN.md exists at `.planning/DESIGN.md` with `schema_version: 1` and all 4 top-level sections
- All sub-headings present: Target Users, Core Problem, Current Alternatives, Pain Points, Primary Emotional Statement, Supporting Attributes, Key Capabilities, Tech Stack, Visual Direction, Brand Personality, Anti-Patterns
- User explicitly approved the DESIGN.md
- Skip path produces no file
- Re-run path offers Update/View/Replace

**Important conventions to follow:**
- All AskUserQuestion headers MUST be 12 characters or fewer
- Each question builds on prior answers -- conversational, not mechanical
- Structured bullet points in DESIGN.md, not prose paragraphs
- The absence of DESIGN.md IS the skip state -- never write empty/placeholder files
- When user selects "Other" on any AskUserQuestion, follow up with plain text, not another AskUserQuestion
